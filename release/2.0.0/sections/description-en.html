Heritage Text Ontology: Description back to ToC

In section introduces the classes and properties used to represent Textual Heritage from the following aspects:
  • Bibliographic Metadata
  • Textual Content
  • Annotation

Bibliographical Metadata

Bibliographic Metadata here mainly contains elements used to identify and retrieve a specific historical textual resource. To be more specific, bibliographic metadata can be described in HTO includes the type (Edition, Series, Volume...) of the heritage resource, title, subtitle, language, genre, printing location, year published. More metadata such as editor, publisher can be expressed by reusing schema.org ontology. Archival structure (e.g. a collection has several editions, a volume has several pages) can be captured using schema:hasPart


An example of using this ontology to represent bibliographic metadata for Encyclopaedia Britannica (EB) collection is shown in Figure 2. In this example, the overall structure of the EB collection is pictured using `schema:hasPart` property. The collection ex:EncyclopaediaBritannica hasPart edition ex:EBEdition7 (the seven edition of EB) and a supplement edition (ex:EBEdition7Supplement). ex:EBEdition7 hasPart volume ex:EBE7Volume2(the second volume of the seven edition of EB). As `schema:hasPart` is used to link two `schema:CreativeWork`, the historical works in this example are `schema:CreativeWork` as well, this can be identified by a reasoner.

Textual Content

Textual content here means the printed or handwritten text in a textual heritage. The knowledge hidden in the textual content is essential to a wide range of researches. However, there are several problems remaining unsolved when it comes to digitalising historical textual documents:

  1. when their content was transcribed either manually or automatically via optical character recognition (OCR) or handwritten text recognition (HTR), text quality of the content varies.
  2. retrieving information on historical documents (such as identifying entity of interest in texts, and semantic relations among entities) is difficult. It faces the challenges of domain heterogeneity, input noisiness, dynamics of language, and lack of resources (such as fine-tuned language models).
To accommodate these problems, this ontology are designed to:
  1. capture textual content extracted from sources provided by different data providers, along with the text quality, provenance information about how the text was extracted and how sources are generated. This allows comparison among text with different qualities as well as tools to generate the text, and possibly provides insight to clean the text.
  2. be extendable and flexible to adopt new entity and relations which would be automatically identified from the textual content in the future.

Annotation

Annotations highlight extra information about resources. Here, we mainly use annotations to connect parts of textual content with associated historical entities (places, persons, events). HTO reuses the Web Annotation Vocabulary for representing annotations.

Figure 5 highlights the key terms used to represent annotations for textual content (OriginalDescription), and we use suffix "oa:" to indicate the terms from Web Annotation Vocabulary. The core class, oa:Annotation, uses oa:hasBody and oa:hasTarget properties to connect resources (bodies and targets respectively), thus showing that a body is related to a target. In this ontology, we mainly use a part of an OriginalDescription as a target, which is an instance of (OriginalDescription). We mainly use an agent (such as a person or organisation) or location as a body. In this case, an annotation can be used to show that a part of a textual content refers to an agent or a location.
